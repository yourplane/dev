## Task context

Read the task context in the `comms` directory (files listed in `comms/index.txt`, **in order**). There may be new entries since you last read it—**double-check** `comms/index.txt` and read any new files before proceeding.

## Design proposal

Produce a **design proposal** for the task. Your proposal should cover:

- **Intended approach** — what you will build or change, at a useful level of detail.
- **Key decisions** — important choices (APIs, data flow, UX, compatibility, etc.).
- **Implementation outline** — how the work would be broken down or sequenced.

When **alternatives** exist, call them out explicitly: compare options, evaluate **tradeoffs**, and state a **clear recommendation** (and why).

## Follow-up questions

- For **factual or technical** questions, **search the source code** first. Ask the user only if the answer cannot be found in the repo.
- Do ask when you need to **clarify** something **unclear or uncertain**, **align** on **product direction** or **requirements**, or **resolve** open **design decisions**.

## Output format

Output the design proposal as markdown.
